# Saleor 3.23 Compatibility Plan

## Support Contract

From this point forward, the configurator targets the current active Saleor minor version.

- The Saleor 3.23 compatibility work ships as configurator `2.0.0`.
- Do not backport Saleor 3.23 compatibility to configurator `1.x`.
- Current target: Saleor `3.23.x`.
- Source of truth: `package.json#saleor.schemaVersion`.
- Patch versions are treated as compatible within the target minor.
- Patch versions are not tracked as configurator support targets.
- Older Saleor minors may still work, but are not actively supported by the latest configurator.
- When Saleor `3.24.0` becomes the active target, configurator parity moves to `3.24.x`.
- We will not add versioned schemas, versioned operations, or compatibility branches per Saleor minor.

## Runtime Version Handling

Networked commands should query the connected Saleor version and warn on minor mismatch.

- Applies to: `introspect`, `diff`, `deploy`.
- Run this as a best-effort preflight before the main command query.
- Use a small query such as `shop { schemaVersion version }`.
- If the compatibility query fails, log/debug it and continue to the main operation.
- Implement this as a shared helper called by the command handlers after GraphQL client construction.
- Keep entity repositories focused on entity operations; do not embed compatibility checks there.
- Does not block execution.
- Does not change exit codes.
- If the operation works on another minor, that is acceptable.
- If GraphQL later fails because the schema is incompatible, that failure remains the real signal.
- In JSON output, the version mismatch should be emitted through the envelope `logs` array as a `warn` entry.
- Do not change per-command `result` shapes for compatibility warnings.

## Work Plan

Each item should be a separate commit and should leave the repository in a coherent state.

1. `docs: plan Saleor 3.23 compatibility`
   - Add this plan and the migration-window policy.
   - No runtime behavior changes.

2. `chore: declare Saleor 3.23 as the supported minor`
   - Change `package.json#saleor.schemaVersion` to `3.23`.
   - Add a small helper for reading/formatting the supported Saleor minor from `package.json`.
   - Do not add runtime warnings yet.

3. `fix(shop): reject and omit Saleor 3.23 removed shop fields`
   - Remove `automaticFulfillmentDigitalProducts`, `defaultDigitalMaxDownloads`, and `defaultDigitalUrlValidDays` from introspection queries and shop update selections.
   - Keep those config keys in Zod as forbidden fields through a shared helper such as `removedInSaleor("3.23")`.
   - Improve validation output for these removed fields.
   - Defensively omit removed fields before `shopSettingsUpdate`.
   - Do not add `useLegacyShippingZoneStockAvailability` yet; this keeps the commit compatible with the old vendored GraphQL schema.

4. `chore(graphql): refresh schema for Saleor 3.23`
   - Refresh vendored GraphQL schema/types using the existing automated generation flow.
   - No hand edits to generated GraphQL files.
   - This comes after removed shop usages are gone so generated 3.23 types do not break existing queries.

5. `feat(shop): support useLegacyShippingZoneStockAvailability`
   - Add the new 3.23 shop setting to config schema, introspection mapping, diff, and deploy input handling.
   - Add focused tests for introspection and deployment shape.
   - This comes after the GraphQL schema refresh because the field exists only in the 3.23 schema.

6. `feat(attributes): support category and collection reference targets`
   - Replace repeated reference target enums with one shared schema/helper.
   - Add `CATEGORY` and `COLLECTION` everywhere reference attributes are accepted.
   - Update validation messages and tests.

7. `feat(cli): warn on Saleor minor mismatch`
   - Add the best-effort `shop { schemaVersion version }` compatibility preflight.
   - Call it from `introspect`, `diff`, and `deploy` after GraphQL client/configurator construction.
   - Emit human-readable warnings in text mode and `warn` log entries in JSON envelopes.
   - Do not change result shapes or exit codes.

8. `docs(schema): publish supported Saleor minor in generated artifacts`
   - Update schema generation to include the supported Saleor minor from `package.json`.
   - Keep removed-field diagnostics out of normal autocomplete/documentation output.
   - Regenerate `schema.json` and `SCHEMA.md`.

9. `chore(release): prepare configurator 2.0.0`
   - Add the Changeset/release note for the breaking support-contract change.
   - Call out that Saleor 3.23 compatibility is not backported to configurator `1.x`.

## Removed Field Policy

Removed-field validation is an upgrade aid, not a compatibility layer.

When Saleor removes a field that existed in the previous configurator target, configurator may keep a temporary targeted validation diagnostic for the migration window. The diagnostic should explain when the field was removed and that it must be deleted from config.

This diagnostic should be removed after the migration window rather than maintained forever.

## Follow-Ups

In a separate PR, add an automated parity check that compares configurator-supported surfaces against the vendored GraphQL schema.

Initial useful checks:

- shop fields queried by introspection exist on `Shop`;
- shop fields sent by deploy exist on `ShopSettingsInput`;
- `AttributeEntityTypeEnum` values in the GraphQL schema match the shared Zod enum.

This should catch schema drift without introducing versioned operation layers.
